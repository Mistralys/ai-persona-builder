# Getting Started

> Output verified against `@mistralys/persona-builder` v2.1.x.

This tutorial walks through building your first pair of VS Code and Claude Code persona files
from scratch. You will end up with two ready-to-use instruction files generated from a single
source.

---

## 1. Install the package

```bash
npm install @mistralys/persona-builder
```

---

## 2. Create the directory structure

Personas live inside a **suite directory**. Each suite has three sub-directories:

```
my-project/
├── persona-build.config.js
├── personas/
│   ├── shared/
│   │   └── partials/               # Cross-suite reusable fragments
│   │       └── greeting.md
│   └── my-suite/                   # Suite source directory (srcDir)
│       ├── meta/
│       │   ├── _shared.yaml        # Suite-wide defaults
│       │   └── code-reviewer.yaml  # Per-persona metadata
│       ├── content/
│       │   └── code-reviewer.md    # Markdown content template
│       └── partials/               # Suite-local reusable fragments
│           └── footer.md
└── dist/
    ├── vscode/                     # Generated VS Code output
    └── claude-code/                # Generated Claude Code output
```

---

## 3. Write the shared partial

**`personas/shared/partials/greeting.md`**

```md
Hello, I am {{name}} v{{version}}.
```

This partial will be included anywhere a template writes `{{> greeting}}`. Variable
substitution (`{{name}}`, `{{version}}`) happens after partial resolution, so the partial
can reference any context variable.

---

## 4. Write the suite-local partial

**`personas/my-suite/partials/footer.md`**

```md
*Generated by AI Persona Builder.*
```

Suite-local partials override shared partials with the same stem name. They are only
available inside the suite that defines them.

> **Partials resolution order (five layers):**
> 1. `BuildConfig.partials` — inline partial map (lowest precedence)
> 2. `sharedPartialsDir` — shared cross-suite partials loaded from disk
> 3. Suite-local partials — files in `<srcDir>/partials/` (this directory)
> 4. `onPartials` plugin hooks — plugin-injected partials (suite-level, highest file-based priority)
> 5. `onPersonaPartials` plugin hooks — per-persona overrides (highest precedence of all)
>
> See [Custom Variables & Dynamic Partials](dynamic-partials.md) for the full reference.

---

## 5. Write the shared defaults

**`personas/my-suite/meta/_shared.yaml`**

```yaml
default_version: '1.0.0'
author: Your Name
cc_permission_mode: default
cc_model: claude-sonnet-4-5
cc_memory: project
```

Every persona in the suite inherits these values unless a per-persona file overrides them.
The `cc_*` fields are required by the default Claude Code frontmatter template. Set them
here once so every persona in the suite picks them up automatically.

See [Metadata Reference → Shared Defaults](metadata-reference.md#shared-defaults--meta_sharedyaml)
for the full list of recognized `_shared.yaml` fields.

---

## 6. Write the per-persona metadata

**`personas/my-suite/meta/code-reviewer.yaml`**

```yaml
slug: code-reviewer
name: Code Reviewer
description: Reviews pull requests and provides structured feedback.
vs_file_name: code-reviewer.agent.md
cc_file_name: code-reviewer.md
tools:
  - read
  - search
show_feedback_template: true
```

Key fields:

| Field | Purpose |
|-------|---------|
| `slug` | Machine identifier; used for cross-suite `{{agent_code_reviewer}}` references |
| `name` | Display name used in frontmatter and templates |
| `description` | Short description used in the default VS Code frontmatter |
| `vs_file_name` | Output filename for the VS Code target |
| `cc_file_name` | Output filename for the Claude Code target |
| `tools` | Tool list serialized into frontmatter by the default templates |
| `show_feedback_template` | Custom boolean flag used by the content template conditional |

---

## 7. Write the content template

**`personas/my-suite/content/code-reviewer.md`**

```md
{{> greeting}}

## Role

{{description}}

{{#if show_feedback_template}}
## Feedback Template

Use this structure for every review:

1. Summary of changes
2. Issues found (severity: high / medium / low)
3. Recommendations
{{else}}
See your team's review guidelines.
{{/if}}

{{> footer}}
```

Processing order:
1. `{{> greeting}}` and `{{> footer}}` are resolved first (partial inclusion).
2. `{{#if show_feedback_template}}` is evaluated (conditional block).
3. `{{name}}`, `{{version}}`, `{{description}}` are substituted (variables).

---

## 8. Create the build config

**`persona-build.config.js`** (ESM, at the project root)

```js
import path from 'node:path';
import { fileURLToPath } from 'node:url';

const __dirname = path.dirname(fileURLToPath(import.meta.url));

export default {
  suites: {
    'my-suite': {
      srcDir: path.join(__dirname, 'personas/my-suite'),
      outputDirs: {
        vscode: path.join(__dirname, 'dist/vscode'),
        'claude-code': path.join(__dirname, 'dist/claude-code'),
      },
    },
  },
  sharedPartialsDir: path.join(__dirname, 'personas/shared/partials'),
};
```

> **CJS alternative:** If your project uses CommonJS, rename the file to
> `persona-build.config.cjs` and replace `import`/`export default` with
> `const path = require('path')` and `module.exports = { … }`.

> **Note:** The deprecated `outVscode`/`outClaudeCode` fields still work for backward compatibility but will be removed in a future major version. Prefer `outputDirs`.

---

## 9. Run the build

```bash
npx persona-build
```

You should see output like:

```
Built 2 persona(s), wrote 2 file(s).
```

---

## 10. Rendered output

The build produces two files.

### VS Code — `dist/vscode/code-reviewer.agent.md`

```md
---
name: 'Code Reviewer v1.0.0'
description: 'Reviews pull requests and provides structured feedback.'
tools: ['read', 'search']
---

Hello, I am Code Reviewer v1.0.0.

## Role

Reviews pull requests and provides structured feedback.

## Feedback Template

Use this structure for every review:

1. Summary of changes
2. Issues found (severity: high / medium / low)
3. Recommendations
*Generated by AI Persona Builder.*
```

Place this file in your VS Code user prompts directory (`.github/copilot-instructions.md`
or your IDE's configured agent instruction folder) to register the persona as an agent.

### Claude Code — `dist/claude-code/code-reviewer.md`

```md
---
name: code-reviewer
permissionMode: default
model: claude-sonnet-4-5
memory: project
allowedTools: ['read', 'search']
---

Hello, I am Code Reviewer v1.0.0.

## Role

Reviews pull requests and provides structured feedback.

## Feedback Template

Use this structure for every review:

1. Summary of changes
2. Issues found (severity: high / medium / low)
3. Recommendations
*Generated by AI Persona Builder.*
```

Place this file in `~/.claude/agents/` (Claude Code's agents directory) to make the
persona available in Claude Code sessions.

## What just happened?

| Feature | Where it appeared |
|---------|------------------|
| Variable substitution | `{{name}}`, `{{version}}`, `{{description}}` resolved from YAML |
| Shared partial | `{{> greeting}}` resolved from `shared/partials/greeting.md` |
| Suite-local partial | `{{> footer}}` resolved from `my-suite/partials/footer.md` |
| Conditional block | `{{#if show_feedback_template}}` rendered the `true` branch |
| Dual-target output | Same source produced both `.agent.md` and `.md` from different frontmatter templates |
| Shared defaults | `cc_permission_mode`, `cc_model`, `cc_memory` came from `_shared.yaml` |

## Next steps

- **Add more personas:** create a new `.yaml` + `.md` pair in `meta/` and `content/`. The build picks them up automatically.
- **Override Claude Code tools:** add a `cc_tools` list to a persona YAML to use a different tool set from `tools`.
- **CI validation:** run `npx persona-build --check --strict` to render without writing and fail the build on any warning or error.
- **Custom frontmatter:** register a plugin with `frontmatterTemplates` to override the default YAML front matter for either target.
- **Cross-suite references:** use `{{agent_code_reviewer}}` in another persona's content to refer to "Code Reviewer v1.0.0" by name.

See [Metadata Reference](metadata-reference.md) for all recognized YAML fields and
[Plugins](plugins.md) for the plugin interface.
